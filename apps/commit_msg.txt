fix(sync): resolve May 8th infinite heal loop & eliminate SQLite WAL corruption

This commit addresses a critical, stacked synchronization failure (the "Zombie Loop") where the React Native JS thread would become trapped in an infinite Forward-Heal retry loop due to underlying Native Android SQLite locking conflicts.

Native Architecture Changes:
- Refactored `BlockerAccessibilityService` to establish a `READONLY` SQLite connection.
- Eliminated `SQLITE_BUSY` (5) locks and prevented WAL corruption (`database disk image is malformed`) caused by the background service fighting the React Native process for `READWRITE` access.

React Native Sync Engine (Circuit Breaker):
- Implemented a Circuit Breaker pattern inside `useTaskActions.ts` Sagas.
- The Heal loop now automatically trips if it hits 15 failed retries (~30 seconds) or detects a `ERR_ACCESS_CLOSED_RESOURCE` (which occurs when the "Nuke & Pave" rebuild wipes the database handle from beneath an active JS loop).

Native Recovery Module (The Nuclear Reset):
- Built a brand new Expo native module: `recovery-module`.
- Implemented `nuclearReset()`, executing `android.os.Process.killProcess()` to forcefully purge corrupted JS memory states.
- Upgraded `HealOverlay.tsx` and `useHealStore.ts` to replace the infinite spinner with an un-dismissible custom `ConfirmationModal` that triggers this native OS-level reboot, allowing the Accessibility Service to auto-restart and the JS engine to grab a fresh SQLite handle upon next launch.
