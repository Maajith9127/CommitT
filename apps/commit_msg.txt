refactor(core/commitments): enforce strict slot overrides and disable autonomous preset harvest

This commit addresses two critical edge cases in the commitment creation pipeline, 
focusing on data integrity and Preset Vault hygiene.

1. Implement "Complete Override Policy" in Frontend Validation
- Updated `taskDraft.ts` to smartly handle Behavioral Rule inheritance.
- Empty time slots now safely inherit the global `verification_style`.
- If a slot contains custom conditions (e.g. a localized GPS limit), it now strictly severs global inheritance and requires an explicit slot-level `ruleId`. This prevents mismatched structural behavior rules from bleeding into custom slots.

2. Deprecate Autonomous Harvesting in Backend Service
- Disabled the `harvestConditionPresets` background loop within the `createInternal` and `updateInternal` Convex services.
- Locations and Digital Blocks are no longer silently pushed to the user's Preset Library on every commit save.
- Added [ARCHITECTURE SHIFT] commentary to document that Preset creation is now an explicit, user-driven action via the Preset Hub to prevent database clutter.
- Verified that the `refreshAccountabilityIdentity` engine (Smart Pre-fill) remains fully operational.
