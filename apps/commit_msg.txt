fix: enforce time-shift reactivity and surgical heartbeat stability

### The Problem
The system was suffering from two conflicting scheduling bugs:
1. **The Stale-Time Stall**: Dragging-and-dropping instances (time-shifting) failed to update the scheduled alarm time because the scheduler was skipping rescheduling if a `scheduled_job_id` already existed (False Idempotency).
2. **The Self-Cancellation Race**: During task failure, the runner would occasionally cancel its own next-slot job while trying to "clean up" the task series.

### The Fix
This commit implements a "Future-Strict Surgical" scheduling model that is both highly reactive to user edits and stable during execution:

- **Enabled Reactive Overrides**: Set `force: true` as the default in `syncTaskSchedule`. This ensures that any change to an instance's `end` time (via UI or API) always forces a job replacement, correctly updating the alarm time.
- **Surgical Discovery Boundary**: Standardized the discovery query to use `gt("end", now)`. This creates a temporal firewall: the scheduler now strictly looks PAST the currently executing slot, making it impossible for a runner to accidentally see or cancel its own session.
- **Aggressive Orphan Purging**: The scheduler now performs a surgical cleanup of any "stray" scheduled jobs ending after the primary next-slot target, ensuring the dashboard remains clean and devoid of duplicate verification jobs.
- **Heartbeat-First Handoff**: Maintained the architectural shift where the next heartbeat is secured at the start of the verification cycle, ensuring chain continuity regardless of how long penalty/waiver initialization takes.

### technical changes
- **scheduler.ts**: Defaulted `force` to `true`; restricted cleanup to future-only sub-queries.
- **runner.ts**: Enforced early sync; look-ahead discovery.
- **api/instances/update.ts**: Explicitly enabled forced sync on every instance update.
