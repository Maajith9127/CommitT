# Engineering Standards & Best Practices

**"The Golden Standard"**

This document establishes the non-negotiable standards for software engineering within this repository. It is the single source of truth for architectural decisions, coding style, and rigorous production practices. When in doubt, or when instructing AI, refer to this document.

---

## 1. Core Philosophy

### 1.1 Readability > Cleverness
Code is read 10x more than it is written.
- **Bad:** One-liners that perform 3 distinct operations to save space.
- **Good:** Verbose, explicit steps that clearly outline the flow of logic.

### 1.2 The "New Employee" Test
Any file you write should be understandable by a new engineer on their first day without them needing to ask you questions. This requires:
- Clear file headers explaining *responsibilities*.
- Comments explaining *why*, not *what*.
- Meaningful variable names.

### 1.3 Modularization & Separation of Concerns
- **Rule of Thumb:** If a file exceeds 300 lines of functional code (excluding imports/types), it is likely doing too much.
- **Layering:**
  - **Data Layer:** Schema definitions, raw types.
  - **Logic Layer:** Pure functions, business rules, validation.
  - **Service Layer:** API endpoints, Mutations, Actions (which call the Logic Layer).
  - **UI Layer:** Components (which call the Service Layer).

---

## 2. Backend Standards (Convex)

### 2.1 The "Secure by Default" Pattern
Every public mutation/query MUST follow this structure:
1.  **Authentication:** Verify the user immediately.
2.  **Authorization:** explicit check (e.g., `task.assigner_id === user.id`).
3.  **Validation:** Call a dedicated validator function (never validate inline for complex logic).
4.  **Execution:** Perform the DB operation.

### 2.2 Validation
- **Centralized Schemas:** Never define Zod or Convex schemas inside a mutation file if they are shared. Move them to `schema.ts`, `shared.ts`, or `lib/validation.ts`.
- **Validation First:** All inputs must be sanitized and validated before any DB reads occur (where possible).

### 2.3 Scheduler Patterns (Chaining)
- **Atomicity:** A chain of events (System Start -> Wait -> Check -> Repeat) must be robust.
- **Independence:** The "Runner" function (e.g., `runScheduledCheck`) should be separate from the "Definition" module (`tasks.ts`).
- **Idempotency:** Scheduled jobs should be safe to retry. Check status before acting.

### 2.4 Modular Architecture
- **Core Domain:** `tasks.ts`, `users.ts` (Handles User Intent).
- **Execution Engine:** `instances.ts` (Handles System Execution/Background Jobs).
- **Intelligence:** `ai.ts` (Handles Volatile/Undeterministic Logic).
- **Shared Lib:** `lib/` (Pure functions, helpers, math, conflict detection).

## 3. Frontend Standards (React Native)

### 3.1 Component Structure
- **Container/View Pattern:**
    - `TaskScreen.tsx`: Handles data fetching, state, and passing props.
    - `TaskView.tsx`: pure UI component, receives data, renders it.
- **Hooks:** Complex logic (more than 10 lines of `useEffect`/`useState`) moves to a custom hook (e.g., `useTaskLogic.ts`).

### 3.2 State Management (Zustand)
- **Selectors:** Always use specific selectors. `const { setTitle } = useStore(state => state)` is bad. `const setTitle = useStore(state => state.setTitle)` is good.
- **Business Logic in Store:** Stores should hold *state* and *actions*. Complex calculations should be in `lib/` functions imported by the store.

### 3.3 Styling
- **Tailwind/NativeWind:** Use utility classes for 95% of styling.
- **Consistency:** Use the theme tokens (colors, spacing) defined in `tailwind.config.js`. Do not hardcode hex values like `#333` in components.

## 4. Coding Style Guide

### 4.1 Naming Conventions
- **Variables:** Nouns (`user`, `pendingTasks`).
- **Booleans:** Prefix with `is`, `has`, `should`, `can` (`isValid`, `hasConflict`).
- **Functions:** Verbs (`validateInput`, `createUser`, `calculateNextSlot`).
- **Event Handlers:** `handle[Event]` (e.g., `handleSubmit`, `handlePress`).
- **Files:**
    - TypeScript: `camelCase.ts` (e.g., `taskValidation.ts`).
    - React Components: `PascalCase.tsx` (e.g., `Button.tsx`).

### 4.2 Comments & Documentation
- **File Headers:** Every file MUST start with a block comment:
  ```typescript
  /**
   * Module Name
   * ==========================
   * Responsibility: What this file does.
   * Key Exports: Main functions.
   */
  ```
- **The "Why":**
  ```typescript
  // BAD: Increment count
  count++;

  // GOOD: Decrement the remaining recurrence count for the NEXT instance
  count--;
  ```

### 4.3 Types
- **No `any`:** `any` is strictly forbidden in production code. Use `unknown` or define the type.
- **Shared Types:** If a type is used in 2+ files, move it to a `types.ts` file.

---

## 5. Production Readiness Checklist

Before merging/deploying, ask:
1.  **Error Handling:** Are we catching errors? Do we show a user-friendly message (`toast`) or just crash?
2.  **Logs:** distinct logs for `[System]` events. Remove `console.log("here")` debug markers.
3.  **Performance:** Are we fetching too much data? Can we use an index?
4.  **Security:** Did I add the Authorization check? Can User A delete User B's stuff?
